# 駅・バス情報ビューア 要件定義書

> **要件定義書とは？**
>
> 要件定義書は設計書よりもさらに **上流** のドキュメントです。
> 「何を作るか」を決める前に「なぜ作るのか」「誰のために作るのか」「どんな課題を解決するのか」を定義します。
>
> **ドキュメントの位置づけ:**
>
> ```
> 要件定義書    →  基本設計書    →  詳細設計書    →  実装
> (なぜ作るか)    (何を作るか)    (どう作るか)    (コードを書く)
> ```
>
> | 観点 | 要件定義書 | 基本設計書 | 詳細設計書 |
> |---|---|---|---|
> | 主な読者 | 発注者・経営者・PM | PM・開発リーダー | 開発者 |
> | 記述レベル | ビジネス・業務の言葉 | 機能とシステムの言葉 | 技術・実装の言葉 |
> | 主な内容 | 目的・課題・要望 | 機能一覧・画面設計 | クラス・関数・DB定義 |
> | 例 | 「どの路線の駅か知りたい」 | 「F-01: 駅・路線一覧表示機能」 | 「GraphQL Queryでstationsとtrainsとbusesをまとめて取得」 |
>
> **なぜ要件定義書が必要か：**
>
> 要件定義がないまま設計・実装を始めると、以下のような問題が起きます。
> - 「思ってたのと違う」 → 手戻りが発生する
> - 「あの機能も欲しい」 → 際限なく機能が増える（スコープクリープ）
> - 「そもそも何のために作ったんだっけ」 → プロジェクトの方向性を見失う

## 1. プロジェクト背景

> **このセクションの目的：** 「このプロジェクトはなぜ始まったのか」を記述する。背景を書くことで、開発中に判断に迷ったとき「そもそもの目的は何だったか」に立ち返ることができる。

### 1.1 背景

東京都の駅情報・バス路線情報・鉄道路線情報を外部サーバーから取得し、それらの関係性をわかりやすく確認したいという要望がある。

従来の地図アプリや乗換案内アプリは多機能すぎて目的の情報にたどり着くのに時間がかかる。また、「ある駅からどのバスが出ているか」「あるバス終点駅にはどの電車が通っているか」という関係を横断的に調べる機能は一般的なアプリには存在しない。

同時に、開発者自身のスキル向上のため、以下の技術を実践的に学習する機会を求めていた。

- Flutter によるモバイルアプリ開発
- GraphQL を使ったデータ取得
- 複雑なデータ加工・マッピング処理の実装
- `scrollable_positioned_list` によるプログラム的スクロール制御

これらの課題と学習目標を同時に満たすプロジェクトとして、本アプリの開発を開始した。

### 1.2 プロジェクト名

**駅・バス情報ビューア (flutter_bus_info)**

## 2. 目的とゴール

> **このセクションの目的：** プロジェクトの成功基準を定義する。「何ができたら完成なのか」が曖昧だと、永遠に終わらないプロジェクトになってしまう。

### 2.1 ビジネス目的

| # | 目的 |
|---|---|
| 1 | 東京都の駅が路線別にまとまって一覧表示される |
| 2 | 各駅に接続するバス路線の終点駅を確認できる |
| 3 | バス終点駅を経由する鉄道路線を素早く確認し、その路線の駅リストへジャンプできる |
| 4 | Flutter・GraphQL・データ加工の実践的な開発スキルを習得する |

### 2.2 達成指標

| # | 指標 | 達成基準 |
|---|---|---|
| 1 | 駅一覧表示 | 東京都の駅が路線ごとにグループ化されて表示される |
| 2 | ナビゲーション | 上部の路線アイコンをタップするとその路線の先頭駅までスクロールされる |
| 3 | バス接続確認 | バス路線を持つ駅を展開すると終点駅の一覧が表示される |
| 4 | 電車情報ダイアログ | バス終点駅の情報ボタンをタップすると、その駅を通る電車の一覧がダイアログで表示される |
| 5 | スクロールジャンプ | ダイアログの電車をタップするとダイアログが閉じ、その電車の先頭駅へスクロールされる |
| 6 | エラー対応 | APIエラー時にリトライボタンが表示され、アプリがクラッシュしない |

## 5. 業務フロー（ユースケース）

> **このセクションの目的：** ユーザーが実際にどのようにアプリを使うかを、具体的なシナリオで描く。

### UC-01: 路線の駅一覧を確認する

```
目的:      特定の路線に属する駅の一覧を素早く表示する
利用者:    エンドユーザー
事前条件:  アプリが起動済みで駅データが取得されている

基本フロー:
  1. ユーザーがアプリを起動する
  2. システムがGraphQLサーバーから駅・バス・路線データを取得する
  3. システムが駅を路線ごとにグループ化してリスト表示する
  4. システムが上部に路線アイコン（CircleAvatar）を横並びで表示する
  5. ユーザーが上部の路線アイコンをタップする
  6. システムがその路線の先頭駅の位置までリストをスクロールする

代替フロー:
  4a. ユーザーがリストを手動でスクロールして目的の路線を探す

例外フロー:
  2a. APIサーバーに接続できない場合
      → エラーメッセージとリトライボタンを表示する
```

### UC-02: 駅に接続するバス路線を確認する

```
目的:      特定の駅から出ているバスの終点駅を確認する
利用者:    エンドユーザー
事前条件:  UC-01が完了し、駅一覧が表示されている

基本フロー:
  1. ユーザーがバスアイコン付きの駅（バス接続あり）をタップする
  2. システムが駅のExpansionTileを展開する
  3. システムがバス終点駅の一覧（件数付き）を表示する
  4. ユーザーが終点駅の情報を確認する

代替フロー:
  1a. バスアイコンのない駅をタップしても展開はされない（ListTile表示のみ）
```

### UC-03: バス終点駅を通る電車を調べる

```
目的:      バス終点駅にどの電車が乗り入れているかを確認し、
           その電車の駅リストへジャンプする
利用者:    エンドユーザー
事前条件:  UC-02が完了し、バス終点駅の一覧が表示されている

基本フロー:
  1. ユーザーがバス終点駅の情報ボタン（ℹ️アイコン）をタップする
  2. システムがダイアログを表示する
  3. ダイアログにその終点駅を通る電車の一覧（路線名・路線番号）が表示される
  4. ユーザーが電車名をタップする
  5. システムがダイアログを閉じる
  6. システムがその電車の先頭駅の位置までリストをスクロールする

代替フロー:
  3a. 電車情報が存在しない場合は「電車情報がありません」と表示する
  7a. ユーザーが「閉じる」ボタンをタップするとジャンプせずにダイアログを閉じる

例外フロー:
  なし（電車情報が空の場合も正常系として処理する）
```

## 6. 機能要件

> **このセクションの目的：** アプリに「何ができてほしいか」を要望レベルで列挙する。基本設計書の機能一覧（F-01〜）の元になるリスト。

### 6.1 要件一覧

| 要件ID | カテゴリ | 要件 | 優先度 | 対応課題 |
|---|---|---|---|---|
| REQ-01 | データ取得 | GraphQLサーバーから駅・バス・路線データをまとめて取得できる | 必須 | P-01, P-02 |
| REQ-02 | 表示 | 駅を路線ごとにグループ化してリスト表示できる | 必須 | P-01 |
| REQ-03 | 表示 | 上部に路線アイコンを横スクロールで表示できる | 必須 | P-04 |
| REQ-04 | 操作 | 路線アイコンをタップするとその路線の先頭駅へスクロールできる | 必須 | P-04 |
| REQ-05 | 表示 | バス接続がある駅を展開してバス終点駅の一覧を表示できる | 必須 | P-02 |
| REQ-06 | 操作 | バス終点駅の情報ボタンをタップすると電車一覧ダイアログが表示される | 必須 | P-03 |
| REQ-07 | 操作 | ダイアログの電車をタップするとその電車の先頭駅へスクロールできる | 必須 | P-03, P-04 |
| REQ-08 | 操作 | 引っ張って更新（Pull to Refresh）でデータを再取得できる | 重要 | P-01 |
| REQ-09 | エラー | APIエラー時にエラーメッセージとリトライボタンを表示できる | 必須 | P-01 |
| REQ-10 | デザイン | 黒メインのダークテーマで表示される | 重要 | P-05 |

### 6.2 優先度の定義

| 優先度 | 定義 | 判断基準 |
|---|---|---|
| 必須 (Must) | この要件が満たされないとアプリとして成立しない | アプリの核となる機能 |
| 重要 (Should) | 可能な限り実装すべき | ユーザー体験を大きく向上させる |
| 任意 (Could) | あれば嬉しい | なくてもアプリとして成立する |

> **補足：** この優先度分類は「MoSCoW法」と呼ばれる手法に基づいている。Must / Should / Could / Won't の頭文字を取ったもの。

## 8. 制約条件

> **このセクションの目的：** 開発における「前提」や「制限」を明記する。

### 8.1 技術的制約

| # | 制約 | 理由 |
|---|---|---|
| 1 | フロントエンドは Flutter で開発する | 学習目的。iOS・Android のクロスプラットフォーム対応 |
| 2 | データ取得は GraphQL を使用する | 外部APIサーバーが GraphQL エンドポイントを提供しているため |
| 3 | キャッシュは HiveStore を使用する | graphql_flutter の推奨するキャッシュ機構 |
| 4 | インデックス指定スクロールは `scrollable_positioned_list` を使用する | Flutterの標準 `ListView` では特定インデックスへのスクロールが困難なため |

### 8.2 外部制約

| # | 制約 | 内容 |
|---|---|---|
| 1 | 駅・バス・路線データ | 外部GraphQLサーバー (`49.212.175.205:8081/graphql`) から取得する |
| 2 | ネットワーク | 初回起動時のデータ取得にはインターネット接続が必要 |
| 3 | 対象都道府県 | APIのクエリで `"東京都"` を指定しているため東京都の駅に限定される |

### 8.3 リソース制約

| # | 制約 | 内容 |
|---|---|---|
| 1 | 開発者 | 1名（個人開発） |
| 2 | 予算 | 無料の範囲（サーバーは既存のものを利用） |
| 3 | 開発ツール | AI（Claude）をペアプログラミング的に活用 |

## 9. リスクと対策

| リスクID | リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|---|
| R-01 | 外部APIサーバーが停止・レスポンスが遅い | 高 | 中 | エラーハンドリングとリトライボタンを実装済み |
| R-02 | 駅データとバスデータの駅名表記が一致しない | 中 | 中 | バスマップは駅名の完全一致で紐付けているため、表記ゆれがあると対応できない |
| R-03 | 路線数・駅数が大量でリストの描画が重くなる | 中 | 低 | `ScrollablePositionedList` の遅延描画が機能するため影響は限定的 |
| R-04 | 半年後にデータ処理ロジックの意味を忘れる | 中 | 高 | 設計書を整備し、コードにコメントを残す |

## 付録A: 用語集

| 用語 | 意味 |
|---|---|
| GraphQL | APIのクエリ言語。RESTと異なり、必要なフィールドだけを指定して取得できる |
| HiveStore | graphql_flutter が使用するローカルキャッシュのストレージ |
| ExpansionTile | タップすると折り畳みが展開されるFlutterのウィジェット |
| CircleAvatar | 円形の画像やテキストを表示するFlutterのウィジェット。本アプリでは路線アイコンとして使用 |
| endA / endB | バス路線の両端の駅名。本アプリでは双方向に紐付けている |
| trainNumber | 路線を識別する番号 |
| Pull to Refresh | リストを下に引っ張るとデータを再取得する操作パターン |
| ScrollablePositionedList | インデックスを指定してアニメーション付きでスクロールできるFlutterパッケージ |
| スコープクリープ | プロジェクトの進行中に範囲が際限なく膨らむこと |
| MoSCoW法 | 優先度分類手法。Must（必須）/ Should（重要）/ Could（任意）/ Won't（対象外）|
